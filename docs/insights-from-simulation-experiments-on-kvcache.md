# LLM KV-Cache 缓存系统仿真实验洞察

---

## 节点数 vs 容量：少节点大容量完胜

**结论**：在总缓存容量相同的情况下，**少节点 + 大容量** 的配置显著优于多节点小容量。

- **案例支撑**：Mountain-Moving中 Pod 从 `64GB` 扩容至 `256GB` 后，缓存命中率与稳定性均有明显提升。
- **原因推测**：
  - 更大的单节点容量降低了驱逐频率；
  - 减少了跨节点缓存碎片化与路由开销；
  - 更利于热点前缀（如 system prompt）长期驻留。

---

## 推演到极致：命中率优化天花板

基于 `trace.jsonl` 的真实请求 trace 数据分析：

| 指标 | 数值 |
| :--- | :--- |
| **总请求数** | 23,608 |
| **总 Block 引用数** | 409,356 |
| **Distinct Block 数** | 183,166 |
| **仅被引用 1 次的 Block** | 143,735（占比 78.5%） |

### 🧮 理论命中率上限计算
$$
\text{Hit Rate}_{\max} = \frac{\text{Total Refs} - \text{Distinct Blocks}}{\text{Total Refs}} = \frac{409356 - 183166}{409356} \approx \mathbf{55.26\%}
$$

### 🔍 关键发现
- **现象**：非连续匹配与前缀匹配命中率完全一致（均为 **55.26%**），无任何损失。
- **原因分析**：
  - Block 的重用始终发生在 **Prompt 前缀位置**；
  - LLM Prefill 阶段具有强顺序性：`hash_ids` 表示 token 序列，共享部分（system prompt、few-shot examples、对话历史）恒居头部；
  - **推论**：一旦某 Block 被缓存，其下次出现时仍位于前缀。不存在“中间 Block 命中但前面未命中”的情况。

### 🧪 实验验证
**场景**：无限缓存 + 单节点（Selector 与 Evictor 失效）

```bash
$ python -m flux_router --data trace.jsonl --nodes 1 --capacity 200000

```

> ✅ **实测命中率 = 55.26%**，与理论值完全一致，验证了该 trace 下缓存潜力已触顶。

## 路由策略的影响（消除驱逐影响）

控制变量：使用 `∞` 容量使驱逐不发生（FIFO ≡ LRU），此时命中率差异完全来自路由策略。

| ∞ 容量 | Random | CacheAware | 纯路由增益 |
|--------|--------|------------|------------|
| FIFO   | 40.15% | 55.26%     | **+15.11pp** |
| LRU    | 40.15% | 55.26%     | **+15.11pp** |

✅ **结论**：
- 纯路由带来的增益为 **+15.11 pp**；
- 驱逐策略在此条件下无影响，验证了“∞ 容量”成功消除了驱逐干扰。

---

## 驱逐策略的影响

控制变量：固定路由策略，比较 LRU 与 FIFO 的差值。

| capacity | Δ(LRU-FIFO) @ Random | Δ(LRU-FIFO) @ CacheAware |
|----------|------------------------|----------------------------|
| 500      | +2.07%                 | +2.21%                     |
| 1,000    | +1.10%                 | +1.31%                     |
| 4,000    | +0.54%                 | +1.60%                     |
| 8,000    | +0.43%                 | +3.59%                     |
| 16,000   | +0.24%                 | +5.21%                     |
| 32,000   | +0.01%                 | **+5.29%**                 |

### 🔍 核心发现
- LRU 对 **Random 路由**几乎无效（最大仅 +2.07pp）；
- LRU 对 **CacheAware 路由**有显著协同效应（峰值 +5.29pp）；
- **原因**：CacheAware 将相同前缀请求集中到同一节点，使得 LRU “最近访问更可能再被访问”的假设成立；而 Random 打散请求，破坏了局部性。

## 核心对比：路由策略 vs 驱逐策略 vs 容量

| 因素 | 变化范围 | 差距 (pp) | 可优化空间 |
| :--- | :--- | :---: | :--- |
| **容量** <br>(cap=8K → ∞, CA+LRU) | 44.77% → 55.26% | **10.49** | 加容量（边际递减） |
| **路由** <br>(Random → CacheAware, ∞容量) | 40.15% → 55.26% | **15.11** | ✅ 改造路由机制 |
| **驱逐** <br>(FIFO → LRU, cap=32K) | 49.53% → 54.82% | **5.29** | 改进驱逐策略 |

### 💡 结论
1. **路由策略是第一决定因素**：带来 **+15.11 pp** 提升；
2. **容量** 居第二位：**+10.49 pp**，但在合理阈值后受成本约束；
3. **驱逐策略影响最小**：仅 **+5.29 pp**，但对 CacheAware 有放大作用。

> 📘 **理论对齐**：此结论与 **Mooncake 论文**一致 —— “调度策略差距远大于驱逐策略”。

## CA+LRU 当前最优策略，但存在热点倾斜问题

簇由首个 `hash_id` 决定（`prefix_match_len` 从左匹配），只要首 id 不同，匹配长度即为 0。因此：
- 簇 0 → 全部路由至 node 0（10,938 请求）
- 簇 46 → node 1（9,203 请求）
- 簇 74 → node 2（3,449 请求）
- 簇 26783 → node 3（18 请求）
- node 4~7 → 完全空闲 ✅ 测试结果符合预期

⚠️ **问题暴露**：
- CacheAware 导致请求高度集中于少数节点；
- 高频簇所在节点缓存压力大，即使 block 被反复访问，仍因 FIFO 或低效驱逐而“误杀”；
- 出现“大量驱逐 → 重新加载 → 再次驱逐”的恶性循环，造成资源浪费。

## 还有一个发现：LRU 为何不需要 Node 4~7？

### 核心对比：LRU vs FIFO 下的行为差异

| 维度 | LRU | FIFO |
|------|-----|------|
| Node 0 缓存满后是否仍能命中前缀 | ✅ 是（100% 请求持续命中） | ❌ 否（第 395 次请求后 system prompt 被驱逐） |
| 最终使用节点数 | 3 个（Node 0/1/2，Node 3 几乎不用） | 8 个全部用满 |
| Node 间缓存重叠 | 0（互补高效） | 有（碎片化浪费） |
| Block 0（system prompt）是否常驻 | ✅ 是 | ❌ 否 |

### 因果链分析

#### LRU 下：
```
热 block（共享前缀）被频繁访问
  → LRU move_to_end → 移至队列尾部
  → 驱逐只赶走队头冷 block
  → 热 block 永不驱逐
  → Node 0 始终保持命中能力
  → selector 总选 hit 最高的 Node 0
  → 不需要路由到 Node 4~7
```

#### FIFO 下：
```
热 block 被频繁访问
  → FIFO 不更新顺序（on_access 无操作）
  → 按插入时间排队
  → 最早插入的共享前缀 block 最先被驱逐（“年龄最大”）
  → Node 0 丢失前缀 → hit=0
  → selector 选择最闲节点（Node 4~7）
  → 新节点重建缓存 → 分散建缓存 → 8 节点全满
```

### 一句话总结
> **LRU 通过保护高频前缀 block，维持了最优路由路径的稳定性，避免请求溢出；FIFO 因无法区分冷热，导致共享前缀被误删，迫使请求分散建缓存，最终形成缓存碎片。**

这也解释了为何 LRU 整体命中率更高（54.82% vs 37.77%）——它将有限槽位集中在关键节点上实现互补，而 FIFO 把资源平摊却造成重复与浪费。

## 还有个发现：链式哈希结构与孤儿 block 的瞬态性

数据证实：每个 block 的前驱唯一，构成**链式哈希（chained hashing）结构**：

```
block_0 (system prompt)
  └─ block_1
      └─ block_2
          └─ block_3 (prefix A ends)
              └─ block_4
          └─ block_5 (prefix B ends)
              └─ block_6
```

这意味着：
- 若 `hash_ids[k]` 在缓存中，则其前驱 `hash_ids[k-1]` 必须也在；
- 因此，**断点之后不可能命中任何 block** —— 因为其父 block 未命中，子 block 不可能存在。

### 更严谨分析：`miss_count = len(hash_ids) - hit_len` 成立的原因

虽然公式看似简单，但过程如下：
1. 处理请求前，可能存在若干“孤儿 block”（gap ≠ 0）；
2. 处理过程中，这些孤儿因无法连接前缀而被驱逐 → 转为 miss；
3. 处理完成后，`miss_count` 包含这些被驱逐项，gap 归零。

👉 **结论**：  
“孤儿 block”仅为**瞬态存在**，在被“发现”前已被本轮处理清除。  
因此 `sum(hits) + sum(misses) == total_needed` 成立，但中间过程并非理想化连续匹配。

## ✅ 优化建议：如何进一步提升 cache 命中率？

### 1. 【首要】推广 Cache-Aware 路由
- 当前 Random 路由命中率仅 40.15%，远低于 CA 的 55.26%
- **行动项**：
  - 以 `hash_ids[0]` 作为一致性哈希 key；
  - 实现 prefix-based cluster grouping；
  - 支持动态扩缩容下的 rebalance。

### 2. 【次优】合理扩容至收益拐点
- 观察显示：当单节点容量 ≥ 32K 时，Random 路由收益趋于饱和；
- **建议**：对 CA+LRU 配置，单节点容量设为 **≥ 32K blocks**，后续扩容性价比极低。

### 3. 【进阶】引入更智能的驱逐策略
- LRU 已有效，但可进一步尝试：
  - **LFU / SLRU**：更好保护高频访问 block；
  - **TinyLFU / W-TinyLFU**：适合长尾访问模式，减少缓存污染。

### 4. 【平衡】缓解热点倾斜：两级路由 + 局部副本
> 目标：既保持高命中率，又避免资源浪费与单点压力。

#### 推荐方案：
- **主从副本机制**：主节点写入，从节点异步复制常用 block（如 system prompt）；
- **Hybrid Routing**：对热点 prefix 使用 round-robin，其余仍按 prefix 路由；
- **动态权重路由**：根据 load 和 hit ratio 动态调整 selector 权重。

### 5. 【工程实践】缓存预加载（Warm-up）
- 对已知高频前缀（如 system prompt），在服务启动时主动加载；
- 避免冷启动首次 miss，提升用户体验。

---

> **最终目标**：在命中率逼近 **55.26%** 上限的同时，实现资源利用率 > 80% 且无单点瓶颈。
